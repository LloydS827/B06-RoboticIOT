# Rerun.io 调研框架与原子能力清单

## 1. 调研目标

本轮调研服务于 Physical AI 数据层项目启动阶段的技术路线判断。核心问题是：Rerun.io 能否成为我们自有 Physical AI 数据层的参考架构、实验底座、封装对象或二次开发基础。

调研不以“Rerun 是什么”为终点，而以“我们后续怎么做”为终点。

## 2. 当前假设

- 用户提到的“在微软的基础上二次开发”，本轮先按上下文理解为“在 Rerun.io 基础上二次开发或封装复用”。
- 第一阶段先做公开资料和源码结构调研，不直接替代后续本地实验。
- 本项目最终目标不是复制 Rerun，而是形成适合机器人、智能工站和制造现场的自有物理数据层。

## 3. 判断标签

| 标签 | 含义 |
| --- | --- |
| 直接借鉴 | 借鉴理念、数据模型、交互范式或产品边界，但不直接使用其代码 |
| 直接复用 | 在原型、实验或早期产品中直接使用 Rerun SDK、Viewer、CLI、RRD 或 Catalog |
| 封装复用 | 在 Rerun 外层建立自有接口、数据规范或适配层，降低后续替换成本 |
| 二次开发 | 基于 Rerun 开源代码、插件/扩展机制或 Rust API 做定制 |
| 自研替代 | 因现场需求、权限、安全、可控性或产品化原因，建议自行建设 |
| 暂缓判断 | 需要实验、源码深入阅读或业务约束确认后才能判断 |

## 4. 调研维度

### 4.1 产品定位与边界

- 是否明确定位为 Physical AI / robotics 数据层；
- 目标用户是算法研发、机器人开发、数据工程、现场调试还是训练团队；
- 覆盖范围是可视化工具、记录工具、数据平台，还是端到端数据闭环；
- 与传统 IoT 平台、SCADA、MES、数据湖、ML dataset 工具的边界。

### 4.2 数据接入与采集

- 支持的 SDK 语言：Python、Rust、C++、Web；
- 支持的数据写入模式：实时 streaming、文件保存、批量列式写入、DataFrame 写入；
- 是否支持多进程、多设备、分布式记录；
- 是否支持外部导入器、MCAP、ROS 2、LeRobot、图像、点云、视频、网格、日志等格式；
- 对机器人控制器、PLC、工业相机、焊机电参、HMI、中台数据的适配可能性。

### 4.3 数据模型

- 实体组织方式：entity path、层级结构、命名规范；
- 数据类型组织方式：archetype、component、datatype；
- 时间组织方式：timeline、event、log_time、log_tick、自定义时间线；
- 空间组织方式：Transform3D、坐标系、frame、相机模型；
- 静态数据、参数、标定、任务上下文如何表达；
- 是否能承载“任务-工件-机器人-传感器-算法-质量结果”的现场业务模型。

### 4.4 存储与数据包

- `.rrd` recording 的语义和边界；
- chunk 的结构、Arrow 列式存储、压缩和索引方式；
- recording、segment、dataset、catalog 的关系；
- 是否适合形成我们自有的数据包格式；
- 是否支持现场敏感数据脱敏、裁剪、分级和版本化。

### 4.5 可视化与回放

- 支持哪些视图：2D、3D、时间序列、文本、表格、图像、点云、轨迹等；
- 是否支持多模态同步回放、时间轴拖拽、事件定位、对比和布局持久化；
- Blueprint 能否沉淀为可复用的场景回放模板；
- Viewer 是否能嵌入自有工具链；
- 对大规模点云、长时序视频、多设备同步回放的性能边界。

### 4.6 查询、整理与训练导出

- 是否支持 DataFrame / SQL 风格查询；
- 是否支持多段 recording 跨数据集查询；
- 是否支持重采样、对齐、多频率传感器融合；
- 是否支持导出为 LeRobot 或其他训练格式；
- 是否支持直接作为 PyTorch DataLoader；
- 是否能支持失败案例库、人工修正、质量反馈和训练样本版本化。

### 4.7 部署、协作与治理

- 开源部分与商业部分的边界；
- 本地单机、局域网、私有化和云端协作方式；
- 用户、权限、审计、数据隔离、敏感数据治理；
- 客户现场部署复杂度；
- 与我们已有项目管理、中台和数据资产系统的集成方式。

### 4.8 二次开发与可控性

- 开源许可证与商业许可风险；
- Rust workspace、Viewer、store、SDK、importer、catalog 的模块边界；
- 是否有稳定扩展 API；
- 自定义 importer、Viewer 扩展、Web Viewer 嵌入、Catalog SDK 的可用性；
- 若未来替换 Rerun，是否能通过自有接口层降低迁移成本。

## 5. 原子能力清单

| 编号 | 原子能力 | 对本项目意义 | 第一轮初判 |
| --- | --- | --- | --- |
| A01 | Python/Rust/C++ SDK 记录数据 | 快速接入算法、机器人应用和实验脚本 | 直接复用 |
| A02 | Entity Path 层级命名 | 表达机器人、相机、工件、任务和算法输出 | 直接借鉴 |
| A03 | Archetype/Component 数据模型 | 统一表达点云、图像、位姿、轨迹、日志和事件 | 直接借鉴，必要时封装 |
| A04 | 多 timeline 机制 | 处理系统时间、传感器时间、机器人 tick、任务阶段 | 直接借鉴 |
| A05 | Transform 与坐标系 | 支撑机器人、相机、工件、世界坐标统一回放 | 直接借鉴，现场规范需自研 |
| A06 | `.rrd` recording | 作为早期实验数据包和回放文件 | 直接复用 |
| A07 | Arrow chunk 存储 | 借鉴列式、多频、多模态数据组织方式 | 直接借鉴 |
| A08 | Viewer 多模态回放 | 研发复盘和样板场景演示 | 直接复用，后续评估二开 |
| A09 | Blueprint 布局 | 固化不同场景的回放模板 | 封装复用 |
| A10 | Catalog server | 多 recording 管理、查询和训练前处理 | 直接复用做实验，产品化暂缓 |
| A11 | Catalog SDK | 数据集整理、过滤、查询和转换 | 封装复用 |
| A12 | DataFusion DataFrame 查询 | 对齐多频数据并导出训练样本 | 直接借鉴/封装复用 |
| A13 | PyTorch dataloader | 从 Rerun 数据直接进入训练流程 | 实验性能力，暂缓产品判断 |
| A14 | LeRobot 导出 | 对接具身智能训练生态 | 直接复用做实验 |
| A15 | MCAP / ROS 2 支持 | 对接机器人常见日志生态 | 直接复用做实验 |
| A16 | External importer | 接入自有现场文件和协议转换 | 二次开发优先入口 |
| A17 | Custom Rust importer | 深度接入自定义格式和 Viewer | 二次开发候选 |
| A18 | Web Viewer 嵌入 | 嵌入自有平台或中台前端 | 暂缓，需验证性能 |
| A19 | Native Viewer Rust 扩展 | 定制现场调试和业务视图 | 二次开发候选，成本较高 |
| A20 | 权限、审计、数据分级 | 客户现场和产品化必需 | 建议自研 |
| A21 | 现场任务/工艺语义模型 | 焊接工站、质量追溯、人工修正 | 建议自研 |
| A22 | 数据版本与样本治理 | 训练、评测、失败案例库 | 借鉴加自研 |
| A23 | 大数据量性能策略 | 长周期、多相机、点云、视频 | 暂缓，必须实验验证 |
| A24 | 部署与私有化策略 | 客户现场可交付性 | 暂缓，需评估 OSS/Hub 边界 |
