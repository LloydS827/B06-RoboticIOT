# Rerun.io 二次开发路线判断矩阵

## 1. 判断口径

本矩阵是第一轮公开资料调研后的初判，用于指导下一阶段实验和源码阅读。它不是最终技术选型结论。

判断维度：

- **业务匹配度**：是否贴合机器人、智能工站、物理数据层；
- **复用价值**：能否缩短样板场景验证周期；
- **二开可行性**：是否有公开 API、扩展机制或清晰源码边界；
- **替换成本**：未来若自研，迁移代价是否可控；
- **产品化风险**：权限、安全、性能、部署、许可证、商业边界。

## 2. 总体路线

建议采用“三层策略”：

1. **实验层直接复用 Rerun**：Python SDK、Viewer、`.rrd`、CLI、OSS catalog、DataFrame 查询、MCAP/LeRobot 工具链。
2. **产品层建立自有抽象**：自有 entity path、任务上下文、工艺 schema、SDK wrapper、importer 规范、数据包结构和导出流程。
3. **关键能力保留替换权**：权限、治理、质量追溯、项目资产、现场部署和长期存储不绑定 Rerun，必要时自研。

这样做的好处是：前期不被基础工具建设拖慢，同时避免后期被 Rerun 数据模型和商业边界锁死。

## 3. 组件判断矩阵

| 模块 | 官方能力证据 | 对本项目价值 | 初步路线 | 主要风险 |
| --- | --- | --- | --- | --- |
| Python SDK | 支持 logging、query、Viewer/CLI 安装 | 快速接入实验脚本、算法模块、数据转换 | 直接复用，外层封装 | 需要避免业务代码直接依赖太深 |
| Rust SDK | Rerun 核心生态语言，Viewer 和底层多为 Rust | 后续高性能采集和深度二开 | 暂缓，源码阅读后判断 | Rust 二开成本高 |
| C++ SDK | 适合机器人控制、视觉和现场程序 | 对接现有机器人/视觉模块 | 实验后封装复用 | C++ 集成复杂度需验证 |
| Viewer | 多模态同步可视化和回放 | 样板场景复盘的最快入口 | 直接复用 | 大数据量性能和业务 UI 边界 |
| Web Viewer | 可嵌入 Web/Jupyter | 可能嵌入自有平台 | 暂缓 | Wasm 32-bit、单线程、内存限制 |
| Blueprint | 保存 Viewer 布局和视图配置 | 场景回放模板 | 封装复用 | 模板治理和跨项目复用需自定规范 |
| Entity Path | 层级组织 entity | 统一表达设备、任务、工件、算法输出 | 直接借鉴 | 命名必须自研规范 |
| Archetype/Component | 内置多模态类型体系 | 统一表达图像、点云、位姿、轨迹等 | 直接借鉴，必要时扩展 | 工艺/质量语义不足 |
| Timeline/Event | 支持多时间线和事件 | 处理多设备、多频率、任务阶段 | 直接借鉴 | 时钟同步和补录规则需自研 |
| Transform | 空间关系和坐标系 | 机器人/相机/工件坐标统一 | 直接借鉴 | 标定版本和工件局部坐标需自研 |
| Static Data | 跨 timeline 数据 | 标定、内参、任务配置、标签上下文 | 直接借鉴 | 静态数据版本管理需自研 |
| `.rrd` | recording 文件 | 早期样本包和回放文件 | 直接复用做实验 | 长期格式锁定风险 |
| Chunk/Arrow | 列式多模态存储 | 借鉴数据包和训练导出设计 | 直接借鉴 | 底层格式细节需源码确认 |
| CLI | 打开、检查、转换、server、mcap、rrd | 工具链和实验自动化 | 直接复用 | 生产自动化需稳定命令验证 |
| OSS Catalog | 本地 catalog server | 多 recording 管理和查询 | 实验复用 | 持久化、协作、权限需验证 |
| Rerun Hub | 商业 catalog/协作/高性能后端 | 产品化参考 | 公开资料调研 | 商业能力不能当作开源可用能力 |
| Catalog SDK | 查询、过滤、操作 catalog | 数据集整理和转换 | 封装复用 | API 稳定性和部署边界 |
| DataFrame Query | DataFusion dataframe 查询 | 多频数据对齐、训练导出、筛选 | 封装复用 | 查询性能需实测 |
| Lenses | 数据抽取、重塑、重路由 | 后处理和派生数据层参考 | 暂缓 | experimental |
| Dataloader | PyTorch Dataset | 直接进入训练链路 | 实验验证 | experimental，大规模建议 Hub |
| LeRobot Export | 导出训练数据集 | 具身智能训练生态对接 | 直接实验 | 我们的标签/质量语义需适配 |
| MCAP | 支持 MCAP 转换和常见消息格式 | ROS/Foxglove 数据生态入口 | 直接实验 | 非 ROS 工业数据仍需 importer |
| External Importer | 独立可执行文件扩展导入 | 现场数据转换优先入口 | 二次开发优先 | 需要建立自有导入规范 |
| Custom Rust Importer | 深度集成 Viewer/SDK 导入 | 高性能/深度定制 | 二开候选 | 成本较高 |
| Native Viewer 扩展 | Rust API 可扩展 UI | 定制现场复盘界面 | 暂缓 | 需要源码和 API 稳定性评估 |
| 权限/审计/治理 | 开源资料未证明完整覆盖 | 产品化和客户现场必需 | 自研 | 涉及安全和交付 |
| 工艺/质量模型 | 非 Rerun 通用能力 | 焊接工站和质量追溯核心 | 自研 | 需要业务深挖 |

## 4. 推荐的二开边界

### 4.1 第一阶段：Rerun 外围封装

先不修改 Rerun 源码，而是在外围建立自有规范：

- `cav_physical_ai` 数据命名规范；
- 标准 entity path；
- 任务上下文 schema；
- 工艺参数 schema；
- 事件和异常 schema；
- Python 记录封装；
- external importer；
- `.rrd` 样本包目录规范；
- README 和实验文档。

这一阶段目标是快速跑通样板场景，同时保留替换空间。

### 4.2 第二阶段：Rerun 扩展与插件化

在确认 Viewer、Catalog、DataFrame 查询能够支撑样板场景后，再考虑：

- 固化 Blueprint；
- 编写自定义 importer；
- 封装 Catalog SDK 查询；
- 构建训练导出工具；
- 验证 Web Viewer 嵌入；
- 评估 Native Viewer 扩展。

这一阶段目标是形成可复用的项目工具链。

### 4.3 第三阶段：自有数据层最小产品

当样板场景验证完成后，判断哪些能力要进入自有 MVP：

- 自有 SDK；
- 自有数据包格式；
- 自有数据治理和权限；
- 自有数据集整理流程；
- 自有训练/评估导出；
- 自有回放模板库；
- Rerun 作为内核、可替换组件，或仅作为参考实现。

## 5. 下一轮实验清单

| 实验 | 验证问题 | 成功标准 |
| --- | --- | --- |
| 最小多模态 recording | Rerun 是否能表达我们核心数据类型 | 图像、点云、位姿、轨迹、日志、事件、工艺参数可同步回放 |
| 多 timeline 实验 | 多设备时间如何表达和查询 | 系统时间、传感器时间、任务阶段可区分并对齐 |
| Transform 实验 | 机器人/相机/工件坐标如何表达 | Viewer 中空间关系正确，标定可复用 |
| Blueprint 实验 | 回放模板能否沉淀 | 重新打开 recording 后布局可复用 |
| Catalog 实验 | 多作业数据如何管理 | 多个 `.rrd` 注册为 dataset/segment 并可查询 |
| DataFrame 查询实验 | 训练样本能否整理 | 可按时间/事件/实体筛选并导出 PyArrow/Pandas |
| LeRobot 导出实验 | 能否进入训练生态 | 能生成最小 LeRobot dataset 或明确缺口 |
| External importer 实验 | 现场文件能否转为 Rerun 数据 | CSV/JSON/图像目录可被 importer 自动加载 |
| 性能冒烟实验 | 点云/图像/视频规模边界 | 记录 CPU/内存/加载时间/回放流畅度 |

## 6. 当前建议

短期不要直接 fork Rerun 做深度修改。更稳妥的路径是：

1. 先用 Rerun 原生能力打通实验；
2. 同时设计自有 schema 和 wrapper；
3. 等真实样板场景跑通后，再判断是否 fork、插件化扩展，或仅把 Rerun 作为开发期工具。

如果一开始就深入 fork，容易在尚未确认现场数据模型前被工具内部结构牵着走。当前更关键的是让真实作业数据被完整记录、回放、查询、整理和导出。
